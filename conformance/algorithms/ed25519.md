# ed25519 keypair (random + deterministic-from-seed)

> **Vectors:** `vectors/ed25519/` *(planned)*
> **Spec:** RFC 8032

## Algorithm

RFC 8032 Ed25519 (PureEdDSA over Curve25519). Two creation paths:

1. Random keypair (CSRNG-generated 32-byte seed).
2. Deterministic from a 32-byte seed (`KeyPair.fromSecret` /
   `KeyPair.generateKeyPairFromSecret`), where the seed is the RFC 8032
   "private key" (input to SHA-512 expansion).

## Inputs / outputs

- Random: side effects only — produces a `KeyPair` instance.
- From-seed: 32-byte seed (base64url string in TS, base64 string in Java —
  both decoded by lenient base64 codecs).
- Output of `export()`: base64url-no-padding string of the 32-byte seed
  (Java actually emits it through `Encoder.encodeToBase64`, which is
  `Base64(0, null, urlSafe=true)` — base64url-no-padding, same as TS).
- Output of `sign(message)`: base64url string of the 64-byte raw R||S
  signature.

## Java impl

- `languages/java/sdk/src/main/java/com/definancy/sdk/crypto/KeyPair.java:37-39`
  `generateKeyPair()` → `new KeyPair((SecureRandom)null)` → uses JCE default
  RNG.
- `KeyPair.java:41-50` `generateKeyPairFromSecret(secret)`: `Encoder.decodeFromBase64`
  of the secret (32 bytes required), then constructs with the seed.
- `KeyPair.java:52-66` `KeyPair(byte[] seed)`: instantiates BouncyCastle's
  Ed25519 generator with a `FixedSecureRandom(seed)` — when BC's
  `Ed25519KeyPairGenerator` calls `random.nextBytes(...)` for 32 bytes,
  it gets the seed back, and derives the public key per RFC 8032.
- `KeyPair.java:128-156` `FixedSecureRandom`: hands out the seed bytes
  exactly once across `nextBytes`; subsequent calls return zero bytes.
  The implementation increments `index` by `bytes.length` (line 147),
  which can over-advance past `fixedValue.length`; later calls return
  early with no data written. Single-use is fine for Ed25519 keygen
  (one 32-byte read).
- `KeyPair.java:84-94` `export()`: parses PKCS#8 X.509 encoding, peels the
  ASN.1 wrapping, reads the inner OCTET STRING (the 32-byte seed),
  encodes via `Encoder.encodeToBase64` (= base64url-no-padding).
- `KeyPair.java:104-113` `sign(message)`: UTF-8 encodes `message`, signs
  with JCE `Signature.getInstance("EdDSA")`, base64url-encodes the 64-byte
  raw signature.

## TS impl

- `ts-definancy-sdk/src/crypto/keypair.ts:46-51` `generate()`: uses
  `crypto.subtle.generateKey("Ed25519", true, ["sign", "verify"])`.
- `keypair.ts:59-97` `fromSecret(secret)`: base64url-decodes the secret
  (must be 32 bytes), wraps it in a hard-coded PKCS#8 byte template
  (`keypair.ts:67-75`) for Ed25519, imports as a private key, then
  derives the public key by exporting the private as JWK and importing
  `{kty, crv, x}` as a verify-only key.
- `keypair.ts:100-105` `export()`: exports PKCS#8, slices the **last 32
  bytes** as the seed, base64url-encodes.
- `keypair.ts:123-127` `sign(message)`: UTF-8 encode via `TextEncoder`,
  sign via `crypto.subtle.sign("Ed25519", privateKey, data)`,
  base64url-encode the raw signature.

## Divergence notes

- **Public key derivation from seed:** RFC 8032 is deterministic. Both BC
  and WebCrypto must produce the same public key for the same seed. Should
  match byte-for-byte.
- **Signature determinism:** Ed25519 signatures are deterministic per
  RFC 8032 (signature = function of `(privateKey, message)`). Both BC and
  WebCrypto must produce identical signatures for the same input. Should
  match byte-for-byte.
- **Secret export encoding:** Java uses
  `Encoder.encodeToBase64` which (urlSafe=true, no padding) is identical
  to TS `encodeBase64Url`. Round-trip should match.
- **Secret import encoding:** Java's `Encoder.decodeFromBase64` accepts
  both `+/` and `-_` alphabets and tolerates padding. TS's
  `decodeBase64Url` only accepts `-_` and depends on `atob` for padding
  handling. A `+/`-encoded secret with `=` padding would round-trip in
  Java only.
- **TS PKCS#8 template (`keypair.ts:67-75`):** hard-coded ASN.1 prefix.
  This is fragile but correct for Ed25519 RFC 8410 PKCS#8 form. If
  WebCrypto changes its accepted PKCS#8 layout, this breaks silently.
- **`FixedSecureRandom` semantics (`KeyPair.java:128-156`):** the index
  is advanced by `bytes.length` rather than `len`, so an over-large
  request leaves the index past the end and subsequent calls return no
  data. For BC's Ed25519 keygen (one 32-byte `nextBytes`) this works,
  but if BC's call pattern ever changes, deterministic-from-seed could
  break in Java without breaking in TS. Should not affect current
  conformance vectors.
- **Sign input encoding:** both UTF-8 encode the message string before
  signing. Same byte-level signing input.
