# sha512_256

> **Vectors:** `vectors/sha512_256/` *(planned)*
> **Spec:** NIST FIPS 180-4 (SHA-512/256)

## Algorithm

FIPS 180-4 SHA-512/256: SHA-512 with a different IV, output truncated to
256 bits. Used for two distinct purposes in the SDKs:

- 4-byte checksum at the tail of a Definancy ID (`id` → SHA-512/256 → last
  4 bytes).
- DPoP `bsh` claim — body integrity hash on requests with a body.

## Inputs / outputs

- Input: arbitrary `byte[]` / `Uint8Array`.
- Output: 32-byte digest.

## Java impl

- `languages/java/sdk/src/main/java/com/definancy/sdk/util/Digester.java:6-12`
  `digest(data)` (alias `sha512_256(data)`) delegates to Apache Commons
  `DigestUtils.sha512_256` — implemented natively via JCE / BouncyCastle.
- Used in `ID.java:58, 78` for checksum, and `DPoP.java:15` for the body
  hash claim.

## TS impl

- `ts-definancy-sdk/src/crypto/digester.ts:5-8` `sha512_256(data)` calls
  `crypto.subtle.digest("SHA-512/256", data)`.

## Divergence notes — **CONFIRMED BUG**

- **The Web Crypto API does not define a `"SHA-512/256"` algorithm.** Per
  the W3C WebCrypto spec, the registered hash names are `SHA-1`, `SHA-256`,
  `SHA-384`, `SHA-512`. Node 18+'s `crypto.subtle.digest` is a WebCrypto
  implementation and rejects `"SHA-512/256"` with a `NotSupportedError`.
  This means **`DefinancyId.toString()`, `DefinancyId.fromString()`,
  `KeyPair.computeId()` (transitively), and `createDpopProof` with a
  non-empty body all throw at runtime in any spec-compliant environment.**
  **Verified via Node REPL — throws `NotSupportedError: Unrecognized algorithm name`.**
  A conforming TS impl will require a third-party SHA-512/256 (e.g.,
  `@noble/hashes/sha512`'s `sha512_256` or a hand-rolled variant).
- Once fixed, byte-level output should match Java's BouncyCastle
  implementation.
